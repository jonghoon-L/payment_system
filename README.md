# payment_system

본 프로젝트는 비즈니스 요구사항이 자주 변경될 수 있는 신규 서비스의 결제 시스템을 구축한 과제입니다. 
유연한 설계(Architecture)와 데이터의 정합성(이력 관리)을 최우선으로 고려하여 구현했습니다.

---

##  핵심 설계 주안점

프로젝트를 진행하며 가장 중요하게 생각한 부분은 **할인 정책이 수시로 추가/변경될 수 있다**는 비즈니스 요구사항에 대비한 객체지향적 설계였습니다. 이를 구현하기 위해 학부 시절 디자인 패턴 수업에서 깊이 있게 학습했던 전략 패턴(Strategy Pattern)과 팩토리 패턴(Factory Pattern)을 결합하여 적용했습니다.


- `전략 패턴`
NORMAL, VIP, VVIP 각 등급별 정책을 독립적인 클래스로 구현했습니다. 또한 PaymentService 내부의 할인 계산 로직을 if-else문으로 하드코딩하지 않고 DiscountPolicy 인터페이스로 추상화하여 책임을 분리했습니다. 
이를 통해 추후 새로운 회원 등급이 추가되거나 할인율이 변동되더라도 기존의 service 코드를 수정할 필요 없이 새로운 정책 클래스만 추가 또는 교체하면 되도록 유연한 구조를 구축했습니다.

- `팩토리 패턴`
 객체 간의 강한 결합을 방지하기 위해 팩토리 패턴을 적용하여 느슨한 결합을 구성했습니다. 또한 service 계층은 factory에게 실제로 어떤 로직으로 구현되는지 알 필요 없이 각 등급에 맞는 정책만 요청하도록 책임을 분리했습니다. 
 이를 통해 코드의 가독성을 높이고, 각 할인 로직에 대해 독립적인 단위 테스트 작성을 할 수 있는 환경을 구성했습니다.

##  브랜치별 설계 전략 및 개선 사항

### 1. `main` 브랜치 (기본 요구사항 - 도메인 주도 설계)
초기에는 인프라(DB)에 의존하지 않고 순수 Java 객체를 활용한 **핵심 비즈니스 로직(결제 및 할인 정책) 구현**에 집중했습니다.
- `MemoryPaymentRepository`를 사용하여 임시 메모리에 데이터를 저장하며 비즈니스 로직을 빠르게 검증했습니다.

### 2. `feature` 브랜치 (심화 요구사항 - 데이터 영속성 및 불변성 부여)
심화 요구사항인 '할인 정책 변경 시 과거 이력 보존'을 구현하기 위해 인프라 계층을 도입하여 설계를 확장했습니다.
- **H2 연동 및 JPA 도입:** 기존의 순수 Java 도메인 객체들을 Spring Data JPA를 활용하여 `@Entity`로 변환했습니다.
- **데이터 정합성 보장 (스냅샷 저장):** 결제 완료 시점에 적용된 할인 정보(등급, 정책명, 할인율, 할인 금액 등)가 `PaymentDiscountHistory` 테이블에 스냅샷 형태로 기록되도록 구성했습니다. 이를 통해 운영 중 할인 정책이 변경되더라도 과거의 결제 영수증은 안전하게 보존되도록 불변성을 보장했습니다.
